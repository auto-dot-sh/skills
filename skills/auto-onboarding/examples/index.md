# Examples

Each example here is a complete `.auto/` directory for one workflow archetype, plus a README explaining the moving parts, what to customize, and how to smoke test it. They are distilled from workflows running in production on the auto repo itself, or modeled directly on the use cases auto pitches on its landing page. Together they cover the platform's whole vocabulary — if you understand all ten, you understand auto.

**Every archetype is also published as a managed template named `@auto/<example-dir-name>`, and creating from the template is the default.** The tenant file is a thin managed import plus the template's variables (listed in each README):

```yaml
imports:
  - "@auto/code-review@latest/agents/pr-review.yaml"
variables:
  repoFullName: acme/widgets
  githubConnection: github-acme
  slackConnection: slack
  slackChannel: "#dev"
```

Identity and avatar come baked into the template — no asset copying — and fields declared in the importing file override the template's, so tailor by overriding (triggers merge by their authoring `name:`; `remove: { triggers: [...], tools: [...] }` drops entries). The templates' shared runtime has no repository setup step: when an agent needs the repo's dependencies installed, override the full inline `environment` with a `setup` block, kept identical across every installed archetype (differing `agent-runtime` definitions conflict at apply). Validate with `mcp__auto__auto_resources_dry_run`, open a PR, and let GitHub Sync deploy after merge. See "Managed templates" in `docs/resource-model.md` for the mechanics.

The directories themselves are the readable source the templates were derived from, and the source to copy from when a workflow needs bespoke authoring no template fits. They differ from the template content in placeholder values and small template-only mechanics (the templates add trigger `name:` keys, mount at `/workspace/repo`, and drop repo-specific environment setup). The placeholder values are what the template variables take: `acme/widgets` for the user's connected `org/repo`, `github-acme` / `slack` / `linear` for their connection names (from `mcp__auto__auto_connections_list`), and `#dev` / `#sales` / `#incidents` for their channels.

Every example agent has a Slack-backed `chat` tool. Agents whose main workflow starts from GitHub, Linear, a webhook, or a heartbeat still include a basic direct-mention trigger so a user can tag the agent, get a short hello, and learn what it does.

For repositories with required GitHub Actions checks, consider adding GitHub Sync CI watchdog workflows during setup with `mcp__auto__auto_sync_enable` and a GitHub `ciWatchdog` config. This is separate from `github.check_run.completed` triggers: triggers react to existing check events, while the watchdog actively dispatches a missing workflow run and should only be configured for dispatchable required Actions workflows.

## The simple (but effective) automations

**`code-review/` — a tailored PR review agent.** On every PR open, reopen, and push, a reviewer wakes with the PR head mounted read-only, reviews the diff against the repo's _own_ written conventions, posts exactly one comment with a thumbs-up/thumbs-down merge recommendation, and reports a GitHub check that can gate merges. A Slack thread per PR carries the verdicts. This is the canonical first workflow: a single spawn trigger, tightly scoped GitHub capabilities (it cannot push, approve, or merge by construction), and an immediately felt payoff. It also gates the agent fleet below.

**`handoff/` — an event-driven coding handoff agent.** A human can mention the agent on a PR or in Slack to hand over implementation, review follow-up, CI failures, or merge conflicts. The handoff records PR ownership, keeps the PR and Slack thread updated, pushes normal follow-up commits, and waits for CI plus the latest PR-reviewer feedback before saying the PR is ready for final review. This is the natural companion to `code-review/`: the reviewer creates the feedback loop, and the handoff agent can use that loop to drive work to readiness.

**`issue-triage/` — ticket triage and resolution.** Two cooperating agents around a Linear label. Adding `auto-triage` to an issue wakes the triage agent, which dedupes, prioritizes, categorizes against _existing_ metadata only, asks for missing context, and — when an issue is implementation-ready — hands off to a spawn-only coder agent via `auto.sessions.spawn` with a structured brief. The coder implements, opens a PR with a Review Map, and reports back on the issue. Demonstrates the label-as-request-token pattern (the agent removes the label after acting, so re-labeling re-requests) and agent-to-agent dispatch.

**`incident-response/` — automated incident and bug response.** Anything that can POST JSON — PagerDuty, a Datadog monitor, Sentry, a runbook `curl` — hits a bearer-token webhook endpoint and wakes a first responder. It correlates the alert with recent commits in the mounted checkout, pulls logs and metrics through an observability MCP tool when one is connected, and posts an evidence-based triage thread to the incidents channel, staying subscribed to answer responder questions. When the evidence points at a clear, contained code fix, it serves the fix up on a platter: a focused branch and a draft PR linked from the triage thread. The hard limits are about authority, not writes: it never merges, restarts services, or declares incidents resolved — humans review the fix and decide. Demonstrates custom webhook triggers, secrets, and remote MCP tools.

**`chat-assistant/` — an @mentionable channel agent.** The smallest complete workflow and the best teaching example: mention the bot, it replies in-thread and subscribes; reply again, and the event delivers back into the _same run_ with its conversational memory intact. This is the purest demonstration of the spawn-vs-deliver attribution pattern (`$.auto.attributions` exists/not-exists on the same event, which validation enforces) and of inline identities — the bot is a real workspace app with its own name and avatar.

**`daily-digest/` — a scheduled reporter.** A cron heartbeat fires every morning; the agent reads the last 24 hours of merged PRs and direct-to-main commits (with timestamps anchored on the heartbeat's scheduled time, so windows stay exact), checks whether the merges actually deployed, and posts a one-sentence summary to the channel with the full digest threaded beneath. Read-only by construction. Demonstrates heartbeat triggers, deeper-history mounts, and output discipline.

## The frontier

**`agent-fleet/` — organized fleets of agents on long-horizon tasks.** The flagship workflow on the auto repo itself. A human tags **@chief** with a list of engineering tasks. The Chief of Staff Engineers — a one-live-session (`concurrency: 1`) orchestrator that never writes code — splits the list into well-scoped tasks, dispatches one staff-engineer session per task with an idempotency-keyed brief, and shepherds the fleet: staff engineers report milestones (`started`, `pr-opened`, `fixing-ci`, `blocked`, `ready`) into the chief's session, PR events route back to each engineer through session bindings, a 15-minute heartbeat sweeps for stalls and respawns dead sessions, ambiguities surface as crisp questions with recommendations, and the batch closes with one collated packet of green, reviewed PRs. Composes with `code-review/` as the quality gate. This example exercises nearly the entire trigger/routing vocabulary — slot-member `deliver` + `onUnmatched: spawn`, `bind` continuation on `github.pull_request`, `attributedSessions` side-threads, heartbeats — plus crash recovery for the orchestrator itself.

**`research-loop/` — multi-agent autoresearch and optimization loops.** Give the research coordinator a measurable objective ("cut p95 latency", "raise the benchmark score") and a budget, and it runs the experimental method: each round it proposes 2-4 falsifiable hypotheses, spawns an isolated experimenter per hypothesis, and collates the measured results — including nulls and negatives — into a lab log posted in the campaign thread. Rounds advance on heartbeats; the loop stops on objective met, budget spent, or two flat rounds; the winning variant becomes a real PR only after explicit human approval. Because the lab log lives in the thread rather than in any run's memory, campaigns survive coordinator restarts. Modeled on the optimization loops we run against our own sandbox-startup benchmarks.

**`lead-engine/` — agentic BDR and outbound lead engines.** The non-engineering archetype, and proof that agents are general agents rather than coding agents: there is no repo mount at all. A new lead arrives by webhook; the agent researches the person and company from public sources, scores the fit with "weak fit, recommend skip" as a first-class verdict, drafts a sub-120-word opener and a follow-up bump, and posts the package to the sales channel for human approval — then revises in-thread on feedback. The hard limit is structural: the agent never contacts a prospect; humans approve and send.

## The loop that closes the loop

**`self-improvement/` — the system learns from real work.** Every couple of hours a self-improvement agent reviews PR feedback, read-only operational or product data sources, and the project's own Auto sessions. It looks for expressed review preferences, repeated friction, failures, anomalies, missed triggers, noisy prompts, and reliability gaps, then reports evidence-backed findings with concrete proposed fixes. Those fixes can target the user's application itself (tests, schemas, docs, error handling, runbooks) or the Auto system (agents, prompts, triggers, routing, process, including the self-improvement agent). Each sweep starts by reading its _previous report_ so it only surfaces what changed, and it prioritizes high-leverage improvements that can be automated going forward. Its own past sessions are in scope.
