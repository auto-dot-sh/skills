# Examples

Each example here is a complete, copyable `.auto/` directory for one workflow archetype, plus a README explaining the moving parts, what to customize, and how to smoke test it. They are distilled from workflows running in production on the auto repo itself, or modeled directly on the use cases auto pitches on its landing page. Together they cover the platform's whole vocabulary — if you understand all nine, you understand auto.

Every example uses the same placeholders, which must be replaced before applying: `acme/widgets` for the user's `org/repo`, `github-acme` / `slack` / `linear` for their connection names (from `auto connections list`), and `#dev` / `#sales` / `#incidents` for their channels. Apply by copying an example's `.auto/` into the repo root, replacing placeholders, then `auto apply --dry-run` followed by `auto apply`.

## The simple (but effective) automations

**`code-review/` — a tailored PR review agent.** On every PR open, reopen, and push, a reviewer wakes with the PR head mounted read-only, reviews the diff against the repo's *own* written conventions, posts exactly one comment with a thumbs-up/thumbs-down merge recommendation, and reports a GitHub check that can gate merges. A Slack thread per PR carries the verdicts. This is the canonical first workflow: a single spawn trigger, tightly scoped GitHub capabilities (it cannot push, approve, or merge by construction), and an immediately felt payoff. It also gates the agent fleet below.

**`issue-triage/` — ticket triage and resolution.** Two cooperating agents around a Linear label. Adding `auto-triage` to an issue wakes the triage agent, which dedupes, prioritizes, categorizes against *existing* metadata only, asks for missing context, and — when an issue is implementation-ready — hands off to a spawn-only coder session via `auto.runs.spawn` with a structured brief. The coder implements, opens a PR with a Review Map, and reports back on the issue. Demonstrates the label-as-request-token pattern (the agent removes the label after acting, so re-labeling re-requests) and agent-to-agent dispatch.

**`incident-response/` — automated incident and bug response.** Anything that can POST JSON — PagerDuty, a Datadog monitor, Sentry, a runbook `curl` — hits a bearer-token webhook endpoint and wakes a first responder. It correlates the alert with recent commits in a read-only checkout, pulls logs and metrics through an observability MCP tool when one is connected, and posts an evidence-based triage thread to the incidents channel, staying subscribed to answer responder questions. Strictly read-only: it never restarts services or declares incidents resolved. Demonstrates custom webhook triggers, secrets, and remote MCP tools.

**`chat-assistant/` — an @mentionable channel agent.** The smallest complete workflow and the best teaching example: mention the bot, it replies in-thread and subscribes; reply again, and the event delivers back into the *same run* with its conversational memory intact. This is the purest demonstration of the spawn-vs-deliver attribution pattern (`$.auto.attributions` exists/not-exists on the same event, which validation enforces) and of session identity — the bot is a real workspace app with its own name and avatar.

**`daily-digest/` — a scheduled reporter.** A cron heartbeat fires every morning; the agent reads the last 24 hours of merged PRs and direct-to-main commits (with timestamps anchored on the heartbeat's scheduled time, so windows stay exact), checks whether the merges actually deployed, and posts a one-sentence summary to the channel with the full digest threaded beneath. Read-only by construction. Demonstrates heartbeat triggers, deeper-history mounts, and output discipline.

## The frontier

**`agent-fleet/` — organized fleets of agents on long-horizon tasks.** The flagship, and the most load-bearing workflow on the auto repo itself. A human tags **@chief** with a list of engineering tasks. The Chief of Staff Engineers — a singleton orchestrator that never writes code — splits the list into well-scoped tasks, dispatches one staff-engineer run per task with an idempotency-keyed brief, and shepherds the fleet: staff engineers report milestones (`started`, `pr-opened`, `fixing-ci`, `blocked`, `ready`) into the chief's run, PR events route back to each engineer through artifact ownership, a 15-minute heartbeat sweeps for stalls and respawns dead runs, ambiguities surface as crisp questions with recommendations, and the batch closes with one collated packet of green, reviewed PRs. Composes with `code-review/` as the quality gate. This example exercises nearly the entire trigger/routing vocabulary — singleton `deliverOrSpawn`, `ownedArtifact` delivery, `attributedRuns` side-threads, heartbeats — plus crash recovery for the orchestrator itself.

**`research-loop/` — multi-agent autoresearch and optimization loops.** Give the research coordinator a measurable objective ("cut p95 latency", "raise the benchmark score") and a budget, and it runs the experimental method: each round it proposes 2-4 falsifiable hypotheses, spawns an isolated experimenter per hypothesis, and collates the measured results — including nulls and negatives — into a lab log posted in the campaign thread. Rounds advance on heartbeats; the loop stops on objective met, budget spent, or two flat rounds; the winning variant becomes a real PR only after explicit human approval. Because the lab log lives in the thread rather than in any run's memory, campaigns survive coordinator restarts. Modeled on the optimization loops we run against our own sandbox-startup benchmarks.

**`lead-engine/` — agentic BDR and outbound lead engines.** The non-engineering archetype, and proof that sessions are general agents rather than coding agents: there is no repo mount at all. A new lead arrives by webhook; the agent researches the person and company from public sources, scores the fit honestly (with "weak fit, recommend skip" as a first-class verdict), drafts a sub-120-word opener and a follow-up bump, and posts the package to the sales channel for human approval — then revises in-thread on feedback. The hard limit is structural: the agent never contacts a prospect; humans approve and send.

## The loop that closes the loop

**`self-improvement/` — the system watches itself.** The workflow Beat 8 of the onboarding installs. Every couple of hours an introspector sweeps the project's own runs using the `auto.runs.*` introspection tools — failures first, then anomalous timing and tool-error patterns — deep-dives at most three, and reports evidence-backed findings with concrete proposed fixes to sessions, prompts, and triggers. Each sweep starts by reading its *previous report* so it only surfaces what changed, and it stays silent when there is nothing actionable. Its own past runs are in scope. Installed as a PR (CI/CD is live by Beat 8), which models the GitOps loop it exists to improve.
